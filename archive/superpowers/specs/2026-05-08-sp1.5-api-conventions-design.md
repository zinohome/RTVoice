### 1.2 Out of Scope

- WS 协议规范化（如统一 ws subprotocol）→ 本 v1 只规定 auth subprotocol；event 命名个别 service 自定，统一留 v2
- 多语言客户端 SDK 自动生成（OpenAPI generator）→ 本 v1 给 hand-written 例子代码即可
- AsyncAPI for WebSockets → 我们采用 markdown 描述 WS，理由见 §3
- Rate limiting 实现 → 仅在 conventions 里规定 headers 格式；实际限流是 SP6 工作
- per-API-key tracking → SP6
- HTTP version migration（HTTP/2 等）→ 看 nginx/Caddy 默认即可，不在 API 规范

### 1.3 关键设计决策

#### D-2026-05-08.1 · 数据面加 `/v1/`，运维面不加

**决策**：
- 加 `/v1/`：`asr` `tts/stream` `tts/stream_ws` `voices` `voices/{id}` `tokens` `sessions` `realtime/{id}`
- 不加：`/health` `/metrics` `/info` `/openapi.json`

**理由**：行业标准（OpenAI / Stripe / GitHub）。运维面与 API 版本无关，加了 `/v1/` 反而 break Prometheus scrape config。

#### D-2026-05-08.2 · HTTP + WS 错误格式同形态

**决策**：所有 4xx/5xx HTTP body 和所有 WS error 事件用同一 schema：
```json
{"type":"error","code":"<machine_code>","message":"<human msg>","request_id":"..."}
```

**理由**：客户端解错误代码逻辑一份。HTTP 无 type 字段也能解析（type 始终 "error" 是冗余但无害）。code 用 `<service>.<reason>` 分层（`tts.voice_not_found`、`stt.invalid_audio`、`session.not_found`）。

#### D-2026-05-08.3 · 立即 hard cutover + 未来软迁移

**决策**：
- 现在：`/asr /tts/* /voices/*` 直接换为 `/v1/...`，老路径返 410。原 agent-worker 同步改 env 默认值，docker-compose 不显式 set 即继承。
- 未来：所有 breaking change 走最少 1 个 release 软迁移：老 endpoint 仍工作 + 加 `Deprecation: true` + `Sunset: <RFC HTTP date>` headers，CHANGELOG 公告，1 release 后 410。

**理由**：当前几乎无外部 consumer（CozyVoice 未接入），hard cutover 成本低；但要给未来 consumers 设定模板。

#### D-2026-05-08.4 · OpenAPI auto-gen + WS 写 markdown

**决策**：HTTP endpoints 走 FastAPI auto-gen `/openapi.json`，每 service 各暴露一份。WS endpoints 在 `docs/api/{service}.md` 里手写 markdown 描述（含 events table）。不引入 AsyncAPI。

**理由**：FastAPI 已用 Pydantic models，OpenAPI 几乎"白送"。WS 端点数少（3 个），AsyncAPI 工具链不成熟，markdown 够用。

#### D-2026-05-08.5 · RESTful collection POST，去除 verb in URL

**决策**：
- `POST /voices/add` → `POST /v1/voices`（创建 voice clone）
- `POST /token` → `POST /v1/tokens`（创建 token，复数化）

**理由**：URL 里不放动词是 REST 惯例。`POST /v1/voices` 与 `GET /v1/voices` 形成 collection 一致，`POST /v1/tokens` 与未来可能的 `GET /v1/tokens` 一致。

#### D-2026-05-08.6 · cursor pagination（即使 v1 没用上）

**决策**：未来 list endpoints 全用 cursor：
```
GET /v1/voices?cursor=<opaque>&limit=50
→ {"data":[...], "next_cursor":"..." | null}
```

**理由**：offset pagination 在 high-throughput insert 下结果不稳；提前定 cursor 避免 v2 时再换。当前 list 数量少（默认 1 个 voice），不分页也能跑。

## 4. OpenAPI Auto-Gen 集成

### 4.1 FastAPI 已有能力

每个 FastAPI app 自动暴露 `/openapi.json`。我们要做：
- 给每个 endpoint 加完整的 `summary` + `description` + `tags`
- 给每个 endpoint 加 `responses={200: ..., 400: ..., 401: ...}` 标注错误返回
- Response models 都用 Pydantic（已经是了）
- request/path/query params 用 Pydantic 或 `Annotated[..., Path/Query/Body]`

### 4.2 实施

```python
# 例：tts-server endpoint
from typing import Annotated
from fastapi import Path, HTTPException

@app.post(
    "/v1/voices",
    summary="Register a new voice clone",
    description="Upload a 16kHz mono wav (3-30s) + reference text. Persists to spk2info.pt.",
    tags=["voices"],
    responses={
        201: {"description": "Voice registered"},
        400: {"description": "Invalid wav or spk_id format", "model": ErrorResponse},
        401: {"description": "Bearer token missing or invalid", "model": ErrorResponse},
        409: {"description": "spk_id already exists", "model": ErrorResponse},
    },
)
async def add_voice(...):
    ...
```

### 4.3 ErrorResponse Pydantic model（统一）

每 service 在 `app/error_schema.py` 各自定义（不引入跨 service 共享目录，避免 docker build context 复杂度）：

```python
from pydantic import BaseModel
from typing import Literal

class ErrorResponse(BaseModel):
    type: Literal["error"] = "error"
    code: str   # e.g. "tts.voice_not_found"
    message: str
    request_id: str | None = None
```

每 service 的 `HTTPException` 改返这个 model 的 dict 形态（`@app.exception_handler(HTTPException)`）。

### 5.3 新增 Pydantic ErrorResponse 处理

每 service 加：

```python
@app.exception_handler(HTTPException)
async def _http_exc(request: Request, exc: HTTPException):
    return JSONResponse(
        status_code=exc.status_code,
        content={
            "type": "error",
            "code": getattr(exc, "error_code", "unknown"),
            "message": str(exc.detail),
            "request_id": request.headers.get("X-Request-ID"),
        }
    )
```

并在 raise 时带 code:
```python
raise HTTPException(status_code=409, detail=f"voice {spk_id!r} 已存在")
# 改为：
exc = HTTPException(status_code=409, detail=f"voice {spk_id!r} 已存在")
exc.error_code = "tts.voice_already_exists"
raise exc
```

或更简洁用自定义 helper：
```python
def api_error(status: int, code: str, message: str) -> HTTPException:
    e = HTTPException(status_code=status, detail=message)
    e.error_code = code
    return e

raise api_error(409, "tts.voice_already_exists", f"voice {spk_id!r} 已存在")
```

implementer 选其一。

### 5.4 实施顺序（无中间 broken state）

1. **先在 server 加新路径**（保留老路径），测试新路径工作
2. **再改 client**（agent-worker / test_page.html）调用新路径
3. **最后删 server 老路径**
4. 每步 rebuild + restart 验证

避免 server 改了 client 没改导致 503 / 404 一段时间。

## 6. Capability Discovery 升级

`/info` 现状（v0.7）返回的字段散乱，统一 schema:

```json
{
  "name": "tts-server",
  "version": "0.8.0",
  "backend": "cosyvoice3",
  "model": "Fun-CosyVoice3-0.5B-2512",
  "capabilities": {
    "streaming": true,
    "voice_clone": true,
    "text_streaming_ws": true,
    "supported_voices": ["default_zh_female", "..."],
    "sample_rate": 24000
  }
}
```

未来加新功能就加 capability key（如 `instruct_supported: true`）。客户端用 `/info.capabilities.X` 探测。
