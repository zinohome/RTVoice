## ⚠️ 未自动验证（受 Playwright 环境限制）

Playwright 的 headless Chromium **没有真实音频设备**：
- `createLocalAudioTrack` 抛 `Requested device not found`
- 即便注入合成音频（sine/white noise），silero VAD 也会拒绝（专门训练过滤非语音）
- 真正的 e2e 必须**真人对着麦克风说话**

未验证的链路：
- VAD speech_start / speech_end 检测
- 状态机转移：Idle → Listening → Thinking → Speaking → Idle
- mock STT/LLM/TTS 流水线触发
- Barge-in：Speaking 中再次说话 → cancel pipeline → Listening
- agent TTS 输出 sine wave 被浏览器播放

## 🧑 人工验证步骤

在有麦克风的机器上：

1. `cp .env.example .env` 并填入随机密钥
2. `./scripts/dev-up.sh`
3. 浏览器打开 `http://127.0.0.1:8000/`
4. 点 **1. 加入房间** → 三个 badge 应转绿（连接 / agent / 远端音频）
5. 点 **2. 开麦克风** → 浏览器要权限，准入
6. 对着麦克风说话约 1-2 秒 → 看 `docker logs -f rtvoice-agent`：
   - 应出现 `[VAD] speech_start`
   - 静音 600ms 后 `[VAD] speech_end`
   - `状态: idle -> listening -> thinking -> speaking -> idle`
   - `[mock STT] -> 你好` 等假转写
   - `[mock LLM] reply=...`
7. 此时**应听到** sine wave 音色的"语音"（agent 的回复）
8. 在 agent 还在"说话"期间，再次对麦克风说话 → 应触发：
   - `[BARGE-IN] 用户打断`
   - `状态: speaking -> interrupted -> listening`
   - sine wave 立刻停止

## 已知缺陷与下一步

- ⚠️ silero VAD 用单线程 onnxruntime，每帧 32ms 推理；CPU 实测延迟需要在生产机收集
- ⚠️ mock TTS 是 sine wave，听感差（v0.4 会换 Kokoro CPU 真 TTS）
- ⚠️ mock LLM 关键词匹配只有 5 句，没有真"对话"
- ⚠️ 状态机退出 INTERRUPTED 时直接 → LISTENING，假设用户"既然打断了肯定要继续说"——这是合理默认但未必所有场景都对
